# Provenance: how the committed calibration numbers were generated

**Read this before citing any number from the calibration report.**

## TL;DR

The committed snapshot and report
([evals/reports/judge_calibration.md](../evals/reports/judge_calibration.md))
were **not** produced by a live LLM. The environment they were authored in had
no API key, so they were generated by a **deterministic simulated stand-in
judge**. Every metric is a *real, correct measurement* — but of the stand-in,
not of a hosted model. To produce numbers for the actual shipped LLM critic,
run one command (below) with an API key.

This file exists so the distinction is never blurred. The committed report also
carries an inline "simulated" banner for the same reason.

## What the simulated judge actually is

`SimulatedJudge` in [evals/judge_calibration.py](../evals/judge_calibration.py)
is a deterministic function with three signals, the first two of which mirror
the production critic's real behavior:

1. **Citation pre-check** *(identical to `CriticAgent`)* — any cited
   `chunk_id` that is not present in the evidence → unfaithful. Catches the
   `fabricated_citation` defects.
2. **Lexical grounding** — for each supporting fact, the fraction of stemmed
   content tokens that also appear in the evidence text. If any fact falls below
   a threshold (0.34) it is flagged unsupported. Catches `unsupported_claim`
   defects (invented surcharges, discounts, requirements — vocabulary absent
   from the evidence).
3. **Decision inversion** — if the evidence carries a refer/decline *directive*
   ("shall be referred", "must decline", …) but the rationale's summary
   concludes the risk is clean (accept language), the rationale has contradicted
   its own evidence → unfaithful. Catches `wrong_conclusion` defects.

**Deliberate blind spot.** The stand-in has *no* signal for `overstated_severity`
— cases where the evidence supports a milder action than the rationale states
while reusing the same vocabulary (e.g. "High SHALL refer" reported as "must be
declined"). Lexical overlap is high and there is no contradiction in conclusion
direction, so the heuristic passes them. This is intentional: it models the
residual fail-open mode a real (but imperfect) LLM judge also exhibits, and it
is what the false-negative rate is meant to surface.

## Why these numbers, not perfect ones

A simulated judge *could* trivially be made to agree with every human label by
peeking at `human_label`. It deliberately does not. It judges only the record
content (rationale + evidence), exactly as a real judge would, so its
disagreements are genuine. The result is a believable, imperfect judge rather
than a circular 100%.

## The committed numbers (simulated backend, N = 24)

| Metric | Value |
| --- | --- |
| Agreement (accuracy) | 83.3% |
| Cohen's kappa | 0.667 |
| Precision (unfaithful) | 83.3% |
| Recall / catch rate | 83.3% |
| F1 | 0.833 |
| False-negative rate (fail-open) | 16.7% |
| False-positive rate (false alarms) | 16.7% |

Confusion matrix (positive = unfaithful): TP = 10, FN = 2, FP = 2, TN = 10.

**Two characterized failure modes:**

- **Fail-open (FN = 2):** both misses are `overstated_severity` — the documented
  blind spot. The judge passes a rationale that escalates a "refer" to a
  "decline" because the words still match the evidence.
- **False alarm (FP = 2):** both are faithful accepts (`JCAL-007`, `JCAL-009`)
  whose evidence *also* states a refer/decline directive for a band or threshold
  that does not apply to this particular risk (roof age 8 < 20; wildfire band is
  Moderate, not High). The inversion signal cannot do the numeric/band reasoning
  to tell that the directive is irrelevant here, so it over-triggers.

Both modes are the kind of finding a real calibration surfaces, and each points
at a concrete fix (add a severity-comparison signal; make the inversion signal
band/threshold aware).

## How to regenerate against the real LLM judge

```bash
LLM_STRUCTURED_OUTPUT_ENABLED=true \
LLM_PROVIDER=openai CRITIC_LLM_MODEL=gpt-4o-mini OPENAI_API_KEY=sk-... \
    python -m evals.judge_calibration --backend llm --record
```

This overwrites the snapshot and report with the live judge's verdicts. The
report header will then read `Judge backend: llm (provider=…, model=…)` and the
"simulated" banner disappears — which is how a reader can tell, at a glance,
whether a given report came from a real model or the stand-in.

The human labels in
[evals/datasets/judge_calibration.jsonl](../evals/datasets/judge_calibration.jsonl)
are fixed; only the judge's predictions change between backends. So the same set
calibrates whichever judge you point at it.
